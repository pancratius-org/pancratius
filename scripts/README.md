# scripts

## import_docx.py

Author-facing importer for one DOCX into the work-bundle layout. It uses
existing Markdown frontmatter under `src/content/**/{ru,en}.md` as the catalog;
it does not read `legacy/data/*` and it does not regenerate the corpus.

New work:

```sh
uv run scripts/import_docx.py /path/to/new.docx --kind book --lang ru
```

Existing work translation:

```sh
uv run scripts/import_docx.py /path/to/book-en.docx --into 30-poslanie-musulmanam --lang en
```

Useful overrides:

```sh
uv run scripts/import_docx.py /path/to/new.docx \
  --kind book \
  --lang ru \
  --number 73 \
  --slug 73-new-work \
  --title "New Work" \
  --description "Short editorial description." \
  --cover /path/to/cover.jpg
```

By default it writes to `src/content/`. For scratch/testing, pass
`--out-content /private/tmp/pancratius-import/src/content`.

The importer creates the bundle folder, writes `<lang>.md`, writes the
cleaned/optimized DOCX artifact to `<lang>.docx`, rewrites imported body images
to `images/<hash>.<ext>`, and writes `bibliography.yaml` when the DOCX contains
a parseable catalog table. Missing descriptions are seeded as an obvious
`TODO:` value that satisfies the content schema.

After import, run the local release-artifact scripts for the changed work before
shipping: render PDF/EPUB and refresh the bulk Markdown archive manifest. CI
only validates and publishes committed artifacts.

## docx_to_md.py

Legacy/batch converter for Pancratius DOCX sources into work-bundle Markdown.
Use `import_docx.py` for adding or updating one author-supplied DOCX in the
current work-bundle layout.

### What it does

For each source `.docx`:

1. Runs `pandoc --from docx --to gfm` with image extraction.
2. Lifts DOCX endmatter bibliography/catalog tables into a
   `bibliography.yaml` sidecar (per `docs/content-model.md`); their cover
   thumbnails never enter the body or the asset pool.
3. Hashes every remaining body image (SHA-256, 12-char prefix) and dedups it
   inside the work bundle at `./images/<hash>.<ext>` relative to that work
   folder. Covers are copied with stable human names (`cover.ru.jpg` /
   `cover.en.jpg`), never hashed.
4. Rewrites body image references to `./images/<hash>.<ext>`.
5. Strips the autogenerated Word TOC block that prefixes many books
   (an optional `# Оглавление` heading plus the run of
   `[Title [pageno](#anchor)](#anchor)` paragraphs).
6. Cleans up cosmetic pandoc artifacts: bold-only headings, setext residue,
   smallcaps and underline spans, `**\**` artifacts, empty headings,
   rights-boilerplate (bounded to non-prose regions), AI alt-text leaks,
   triple+ blank lines.
7. Extracts cross-references from footnotes and inline mentions into
   frontmatter `cross_refs` (resolved-only, `{target: {kind, number}, …}`).
8. Concatenates multi-part books (e.g. `02-1`, `02-2`, `02-3`) into one MD
   per language under separating `## Часть N` headers. For book 71 the EN
   `-clean.docx` variant is preferred over `-pre-cleanup*` drafts.
9. Emits each book/poem/project to its work-bundle folder with YAML
   frontmatter Astro content collections consume directly.
10. Updates `data/conversion-manifest.json`. Each run records the source DOCX
    files used for each work and every file the converter wrote per work in
    `by_work[<kind/slug>].generated_paths`
    (a dict keyed by owner — this script writes under `"docx_to_md"`,
    `docx_optimize.py` writes under `"docx_optimize"`). Paths are relative
    to the work folder, so the manifest is portable across `--out-content`
    scratch trees. A rerun deletes only stale paths from its own owner slot;
    other owners' slots and any unknown author-added neighbors are preserved.

Slugs, numbers, titles, tags, and covers are read from
`legacy/data/library-data.js` / `poetry-data.js` / `projects-data.js` — the
new site's URLs match what the legacy site already implied.

### Output layout

```
src/content/
  books/<ascii-slug>/
    ru.md
    en.md                 # when an EN translation exists
    cover.ru.jpg          # stable, human-named
    cover.en.jpg          # optional, when EN exists
    images/
      <hash>.<ext>        # converter-imported body images (dedup'd within work)
      <named>.<ext>       # author-named body assets (preserved across reruns)
    bibliography.yaml     # optional catalog snapshot sidecar
    meta.json             # conversion-time provenance (not UI truth)
  poetry/<ascii-slug>/ru.md (+ optional cover.ru.<ext>)
  projects/<ascii-slug>/{ru,en}.md (+ optional cover.ru.<ext>)
data/
  conversion-manifest.json    # {stats, by_work, by_hash}
```

`by_work[<kind>/<slug>]` looks like:

```jsonc
{
  "kind": "book",
  "slug": "01-evangelie-tsarstviya",
  "generated_paths": {
    "docx_to_md": [
      "bibliography.yaml",
      "cover.en.jpg",
      "cover.ru.jpg",
      "en.md",
      "images/649a499a5bdb.jpg",
      "images/9b31c57123fd.jpg",
      "meta.json",
      "ru.md"
    ],
    "docx_optimize": ["en.docx", "ru.docx"]
  },
  "sources": {
    "ru": [{ "path": "legacy/books/ru/01-евангелие-царствия.docx", "filename": "01-евангелие-царствия.docx" }],
    "en": [{ "path": "legacy/books/en/01-евангелие-царствия.docx", "filename": "01-евангелие-царствия.docx" }]
  },
  "images": [{ "image_index": 1, "hash": "...", "ext": ".jpg", "role": "body", ... }]
}
```

### Frontmatter

Books:

```yaml
---
kind: book
number: 33
slug: 33-ya-esm-vsadnik-kon-i-mech
title: Я Есмь — Всадник, Конь и Меч
lang: ru
description: …                  # mandatory; SEO / OG / card copy
tags: [Откровение Бога, Библия]
cover: ./cover.ru.jpg           # relative to the work folder
translation:
  source: original
---
```

Poems add `date`; projects use `kind: project`. Cross-language pairing is
`(kind, number)`, the single corpus-wide identity rule.

### Requirements

- `pandoc` >= 3.x on `PATH` (`brew install pandoc`)
- `uv` with the repo `pyproject.toml` / `uv.lock` environment.

### Usage

Convert selected legacy source groups:

```sh
uv run scripts/docx_to_md.py --kind book --kind poem
uv run scripts/docx_to_md.py --kind project
uv run scripts/docx_to_md.py --kind project --slug enlightened-ai
```

Just the small test set into `.cache/converter-test/` (no mutation of the real
site directories):

```sh
uv run scripts/docx_to_md.py --test
```

One book or one poem by number:

```sh
uv run scripts/docx_to_md.py --kind book --number 33
uv run scripts/docx_to_md.py --kind poem --number 2
```

Destructive clean-room rebuild for the selected works only. `--clean` removes
each selected work bundle before regenerating it; it does not wipe whole content
kind directories and does not touch unselected works.

```sh
uv run scripts/docx_to_md.py --kind book --kind poem --clean
uv run scripts/docx_to_md.py --kind project --clean
uv run scripts/docx_to_md.py --kind project --slug enlightened-ai --clean
```

Custom output paths (e.g. scratch output outside the repo):

```sh
uv run scripts/docx_to_md.py --kind book --kind poem \
  --out-content /tmp/pancratius-scratch/src/content \
  --manifest /tmp/pancratius-scratch/conversion-manifest.json
```

### Idempotency

Hashes are content-addressed and `generated_paths` are work-folder-relative,
so re-running on the same inputs produces the same paths, the same dedup'd
images, and an identical manifest. `stale files removed: 0` is the
"nothing changed" signal.

### Performance

Full corpus (72 books × ru/en + 43 poems + 2 projects, ~1.2 GB of inputs) runs
in well under a minute on an M-series Mac. Pandoc is the bottleneck.

### Known gotchas

- Pandoc escapes `|` and `>` as `\|` / `\>` inside paragraphs (so a stray
  pipe doesn't get parsed as a GFM table cell). Markdown renderers render
  these as the literal characters; nothing to do.
- The first `![image1.png]` style image in some docx is sometimes the cover
  repeated inside the body. The canonical cover remains `cover.<lang>.<ext>`;
  if the repeated inline image is kept, it is treated like any other body
  asset and referenced as `./images/<hash>.<ext>` inside the work bundle.
- Imported body illustrations use stable work-local paths:
  `![](./images/<hash>.<ext>)`.
- A few books embed Russian-language alt text on AI-generated images — kept
  as-is on the rewritten Markdown image alt text.
- For book 71 we pick the EN `-clean.docx` over the `-pre-cleanup` drafts.
  If you ever want to compare versions, run with `--kind book --number 71` after editing
  `_pick_en_files`.

### Re-running just a subset after edits

Each run writes only the files for the works it converts. To regenerate one
book without disturbing others:

```sh
uv run scripts/docx_to_md.py --kind book --number 33
```

Partial runs carry forward `by_work` and `by_hash` entries from the previous
manifest for works they didn't touch, then recompute corpus-wide stats from
the merged view — so `data/conversion-manifest.json` stays internally
consistent regardless of how narrow a partial run is.

## docx_optimize.py

Shrinks the original `.docx` files for distribution as downloadable Word
documents. Source/destination mapping comes from
`data/conversion-manifest.json` provenance, not content frontmatter. The corpus
carries ~95% of its bytes as uncompressed 1024×1536 RGB PNGs from an AI image
generator; this script re-encodes those as JPEG q85 (visually
indistinguishable on phones/tablets) and rezips the result.

Full corpus: **1.1 GB → 267 MB (~77% saved)** in ~35 s on an M-series Mac.
The 65 MB outlier drops to 7.8 MB.

### Strategy

For each image in `word/media/`:

| Input                                                | Action                    |
|------------------------------------------------------|---------------------------|
| Tiny icon (< 30 KB and ≤ 256 px)                     | leave alone               |
| PNG, no alpha, large (≥ 512 px or ≥ 200 KB), smooth  | **JPEG q85 progressive**, rename `.png` → `.jpg` |
| PNG with alpha, palette, or sharp lines (high edge density) | re-save as optimized PNG |
| Existing JPEG, < 200 KB                              | leave alone               |
| Existing JPEG, ≥ 200 KB                              | re-encode q85 if it wins ≥ 5% |
| Long edge > 1600 px                                  | LANCZOS-downscale to 1600 first |

When an extension changes, the script rewrites the `Target=` attribute in
every `word/**/*.rels` and ensures `[Content_Types].xml` declares a
`<Default Extension>` for the new extension. Other parts of the docx
(`document.xml`, styles, headers, etc.) are copied verbatim.

### Output layout

Optimized source files land alongside their markdown siblings in the content
tree — not under `dist/`. Multi-source books keep part-N suffixes as source
artifacts; everything else is single-file `<lang>.docx`:

```
src/content/books/<slug>/{ru,en}.docx                    # single-source books
src/content/books/<slug>/{ru,en}-part{1,2,3}.docx        # multi-source books (02, 46)
src/content/poetry/<slug>/ru.docx
src/content/projects/<slug>/ru.docx
```

For a merged multi-source work, the public `<lang>.docx` download is produced
separately by `render_downloads.py --docx` from the merged Markdown. Do not
expose a part file as the per-work DOCX download.

The destination is computed from the content tree: `meta.json` for books
(canonical multi-part filename list), markdown frontmatter for poems and
projects. Originals under `legacy/` are read-only.

### Usage

```sh
# Full corpus (skips outputs newer than their source).
uv run scripts/docx_optimize.py

# Force rebuild + per-image stats.
uv run scripts/docx_optimize.py --force --verbose

# Specific file(s) or directory; destination is still derived from src/content/.
uv run scripts/docx_optimize.py "legacy/books/ru/53-голос-еноха-...docx"

# Dry run.
uv run scripts/docx_optimize.py --dry-run
```

### Verification

```sh
unzip -t src/content/books/<slug>/ru.docx
textutil -info src/content/books/<slug>/ru.docx
qlmanage -t -s 1800 -o /tmp src/content/books/<slug>/ru.docx
```

### Tunables

Constants at the top of `docx_optimize.py`:

| Name | Default | Effect |
|------|---------|--------|
| `MAX_LONG_EDGE` | 1600 | Downscale anything bigger |
| `JPEG_QUALITY` | 85 | Subjectively transparent at ≤ 1600 px |
| `LARGE_PNG_MIN_EDGE` | 512 | Below this → keep PNG |
| `LARGE_PNG_MIN_BYTES` | 200_000 | Below this → keep PNG |
| `EDGE_DENSITY_DIAGRAM_MIN` | 0.13 | Above this → keep PNG |

### Known limitations

- Validated against macOS Quick Look + `textutil` (Apple's Office Open XML
  parser used by Pages/TextEdit/Preview). LibreOffice was not installed in
  this environment.
- No `mozjpeg`; we use Pillow's default JPEG encoder.
- Re-encoding PNG → JPEG drops any alpha. The heuristic only fires on PNGs
  without alpha, so this is safe; but if a future Word-emitted PNG carries
  alpha in a "photographic" image it will be kept as PNG.
- `dist/` is regenerable output — do not commit.

## render_downloads.py

Local admin tool that renders release artefacts into the work bundle, per the
contract in `docs/downloads.md`. PDF/EPUB outputs land as
`src/content/<kind>/<work>/<lang>.{pdf,epub}`. Merged multi-part DOCX outputs land
as `src/content/books/<work>/<lang>.docx` when `--docx` is passed. These files are
**committed**; CI never runs this script and never invokes pandoc or typst.

### Requirements

- `pandoc` (>= 3.0) on `PATH`
- `typst` (>= 0.13) on `PATH`
- `uv`

The Cyrillic-capable fonts and the typst template live in
`scripts/downloads-templates/` and `scripts/downloads-fonts/`. Both are
committed; no network access required.

### Usage

```sh
# Everything (cold run ≈ 1m40 on M-series for 148 PDFs + 101 EPUBs).
uv run scripts/render_downloads.py

# One work.
uv run scripts/render_downloads.py --book 33
uv run scripts/render_downloads.py --poem 1
uv run scripts/render_downloads.py --project enlightened-ai

# One language only.
uv run scripts/render_downloads.py --lang en

# Selective formats.
uv run scripts/render_downloads.py --skip-pdf
uv run scripts/render_downloads.py --skip-epub

# Merged DOCX release artifact for a multi-part work.
uv run scripts/render_downloads.py --book 2 --docx --skip-pdf --skip-epub --force

# Force regeneration even if the output is newer than the source.
uv run scripts/render_downloads.py --force
```

### Output policy

EPUB is books-only per `docs/downloads.md` scope. Poems and projects skip EPUB.

The script is incremental by default — it skips outputs newer than their
source `<lang>.md`. Use `--force` to rebuild everything.
